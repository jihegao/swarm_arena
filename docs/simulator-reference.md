# Swarm Arena 教学仿真参考文档

本文档总结 `swarm_arena` 的项目目标、设计思路和学生实验过程，供 `Simulator` 项目参考。它关注的是可迁移的教学仿真模式：如何让学生用 vibe coding 创造一个可运行对象，观察其行为，再和老师讨论下一轮升级目标。

## 项目目标

`swarm_arena` 是一个面向课堂的竞争生存仿真项目。学生编写自己的 `Creature`，把它放进一个 2D 世界中，与内置物种和其他学生物种竞争。世界中会随机生成食物，生物通过移动、进食、捕食和繁殖争夺生存空间。到 5000 tick 时，存活数量最多的物种获胜；如果只剩一种物种，游戏提前结束。

项目的核心教学目标不是让学生一次写出最强 AI，而是让他们经历一个清晰的创造循环：

1. 用自然语言和少量代码提出一个生物设想。
2. 通过 vibe coding 快速生成或修改 `decide()` 策略。
3. 把生物放入同一个仿真世界观察表现。
4. 根据失败原因、成功行为和对手策略，讨论下一轮升级目标。
5. 重复迭代，让学生逐步理解感知、行动、反馈、约束和策略权衡。

这个项目适合承载“学生创造自己的智能体”这一类课程，因为成功标准是可视、即时、可比较的。学生不用先理解完整引擎，只需要理解自己能看到什么、能做什么、怎样判断下一步。

## 教学设计原则

### 降低创作入口

学生只需要在 `creatures/` 目录下新增一个 Python 文件，定义一个继承自 `Creature` 的类，并实现 `decide(perception) -> Action`。引擎会自动加载合法类并生成初始个体。

这种入口有三个好处：

1. 学生的创作边界明确：只改自己的 creature 文件。
2. 老师可以给出最小模板，学生能立即运行。
3. 项目能同时容纳多个学生作品，形成自然比较和讨论。

给 `Simulator` 的参考：如果目标也是让学生创造对象，应尽量提供一个“单文件、单类、单入口函数”的创作面。引擎负责世界状态和规则，学生负责策略。

### 把复杂系统压缩成感知和动作

`swarm_arena` 不让学生直接读写全局世界，而是通过 `Perception` 给每个生物有限视野：

- 自身位置、朝向、能量、体型。
- 视野内食物的位置、大小、能量值。
- 视野内其他生物的位置、类型、体型、能量比例。

学生返回一个 `Action`：

- 朝哪个角度移动。
- 用多快速度移动。
- 是否声明追击目标。
- 是否请求繁殖。

这让策略问题变成一个小而清楚的闭环：看见局面，做出动作，观察结果。

给 `Simulator` 的参考：不要一开始暴露完整仿真内部状态。给学生一个稳定、有限、可解释的观察对象，再给一个小动作集合。这样学生可以用自然语言描述策略，LLM 也更容易生成可维护代码。

### 用规则创造取舍

项目里的属性系统限制生物不能全项拉满。速度、初始能量、视野半径都有上下限，并共享一个点数预算。高速度带来更强机动性，但移动和大体型都会增加能量压力；高能量带来初始体型优势，但也可能成为更大生物的目标；大视野能更早发现食物和威胁，但会占用点数预算。

这些规则让课堂讨论从“怎么作弊变强”转为“你想成为什么类型的生物”：

- 快速抢食型。
- 稳健觅食型。
- 捕食滚雪球型。
- 低风险繁殖型。
- 团队/群体策略型。

给 `Simulator` 的参考：教学仿真最好有明确资源约束和策略取舍。学生的升级目标应该有代价，否则讨论会退化成“把所有数字调大”。

### 让失败可观察

生物可能输在不同原因上：

- 没抢到食物，能量耗尽。
- 追击太冒险，被更大生物吃掉。
- 繁殖太早，后代太小。
- 繁殖太晚，数量不够。
- 视野太小，错过机会。
- 速度太慢，无法脱离威胁。

这些失败原因都可以映射到下一轮升级目标。例如：

- “先别追猎，低能量时优先找食物。”
- “遇到比自己大 10% 的生物就逃跑。”
- “能量超过 2.5 倍再繁殖，而不是刚到阈值就繁殖。”
- “边界附近容易被围住，加入远离边界逻辑。”

给 `Simulator` 的参考：仿真最好能产生可解释的失败现象。学生和老师讨论升级目标时，最好能从行为观察出发，而不是只看最终分数。

## 系统设计思路

### 分层结构

项目采用轻量分层：

```text
main.py               # pygame 主循环和按键操作
world.py              # 仿真 tick、实体管理、感知构建、规则结算
creature.py           # Creature / Perception / Action 公共 API
combat.py             # 大吃小规则
food.py               # 食物实体
renderer.py           # 可视化和排行榜
creature_loader.py    # 自动加载学生和内置生物
creatures/            # 内置生物和学生生物
training/             # GA/RL 等训练实验
```

关键边界是：`World` 拥有真实状态和规则，`Creature` 只通过 `decide()` 表达意图。移动、进食、捕食、繁殖、能量扣除都由 `World` 统一结算，避免学生代码直接破坏世界。

给 `Simulator` 的参考：学生扩展点应该表达“意图”，而不是直接修改仿真状态。这样能保留公平性、可测试性和课堂可控性。

### 安全护栏

为了适配 vibe coding，项目需要把学生代码视为不稳定输入。当前实现包含几类护栏：

- `decide()` 抛异常时，世界不会崩溃，该生物本帧停止，并打印 warning。
- 返回值不是合法 `Action` 时，会使用停止动作兜底。
- 非法角度、速度、NaN、inf 会被清洗。
- 一个文件里暴露多个合法 Creature 类时，会打印 warning，提醒旧实验类可能意外参赛。
- 类名和 `creature_type` 不一致时，繁殖仍能按实际类型生成后代。

给 `Simulator` 的参考：只要学生用 LLM 生成代码，就必须有运行时兜底。课堂上最伤节奏的不是策略失败，而是一个学生的错误代码让整个仿真停掉。

### 内置对手作为脚手架

项目内置多种行为鲜明的生物：

- `Hunter`：追猎弱者，能量低时找食物。
- `Grazer`：以食物为主，近距离逃跑。
- `PackHunter`：尝试靠同伴聚集形成优势。
- `Scavenger`：速度快，抢食物，不主动攻击。

这些对手不是为了“最优”，而是为了提供可讨论的生态位。学生可以针对某个对手设计克制策略，也可以模仿某个对手再改出自己的风格。

给 `Simulator` 的参考：内置样例最好不是一个万能 AI，而是一组性格清楚、弱点明显的 baseline。这样学生更容易提出“我的版本要比它好在哪里”。

### 可视化反馈

可视化界面承担课堂反馈功能：

- 主区域显示生物、食物和移动方向。
- 顶部显示 tick 和各物种存活数量。
- 侧边栏显示种群排行榜和能量最高的个体。
- 游戏结束时显示获胜物种和数量对比。

这些信息足够支持第一轮讨论，但后续可以继续加强事件解释，比如统计吃食物次数、捕食次数、繁殖次数、常见死因。

给 `Simulator` 的参考：可视化不只是好看，它是讨论媒介。界面上应该显示能帮助学生提出下一轮假设的信息。

## 学生实验流程

### 第 1 步：运行基准世界

老师先运行：

```bash
python main.py
```

学生观察内置物种如何竞争：谁先吃食物，谁会追击，谁容易死，最后谁靠数量获胜。此时重点不是写代码，而是建立规则直觉。

讨论问题：

- 哪种生物看起来最危险？
- 哪种生物前期弱但后期可能变强？
- 食物、体型、繁殖之间有什么关系？

### 第 2 步：创建最小生物

学生复制最小模板，创建一个只会直线移动的生物。这个阶段的目标是跑通提交路径：

```text
creatures/my_creature.py
```

成功标准：

- 游戏启动时能看到自己的物种。
- 排行榜里出现自己的类型名。
- 代码出错时能根据 warning 定位问题。

### 第 3 步：第一次策略升级

学生给自己的生物添加一个简单目标，例如：

- 看见食物就朝最近食物移动。
- 看见更大生物就逃跑。
- 能量超过阈值就繁殖。

这一轮只改一个策略点，避免一次性生成太多逻辑，无法判断哪个修改有效。

老师可以引导学生把自然语言目标写成清楚的规则：

```text
当我能量低于 50% 时，优先找最近食物。
当 60px 内出现比我大的生物时，立刻朝反方向逃跑。
当我能量超过 max_energy * 2.5 且附近没有大敌人时，繁殖。
```

### 第 4 步：观察并记录表现

运行游戏后，学生记录：

- 大概第几 tick 死亡或占优。
- 是因为饿死、被吃、繁殖失败，还是数量不足。
- 是否出现预期行为。
- 是否有意外行为，比如绕圈、撞墙、追错目标。

这一步非常重要。没有观察记录，vibe coding 容易变成随机改代码。

### 第 5 步：讨论下一轮升级目标

老师和学生一起把观察转成下一轮目标。建议目标格式：

```text
现象：我的生物会追小目标，但经常追到大敌人附近被吃。
假设：追击前应该检查附近是否有更大威胁。
升级目标：只有当 80px 内没有更大生物时，才追击猎物。
验证方式：观察前 1000 tick 内被大生物吃掉的次数是否减少。
```

这种格式适合学生和 LLM 协作，因为它把“改得更聪明”变成一个具体、可测试的小改动。

### 第 6 步：可选训练实验

当学生已经写出可运行规则后，可以进入训练实验：

- GA：优化可调参数，比如速度、视野、繁殖阈值、逃跑距离。
- RL：学习在不同局面下选择高层动作。

训练不替代手写策略，而是作为进阶讨论：哪些参数适合机器搜索，哪些规则仍需要人类解释和设计？

## 推荐课堂节奏

### 45 分钟短课

1. 5 分钟：展示内置物种对战。
2. 8 分钟：学生创建最小生物并跑通。
3. 12 分钟：添加找食物或逃跑策略。
4. 10 分钟：观察结果并记录问题。
5. 10 分钟：讨论下一轮升级目标，每人写出一个清楚的规则假设。

### 90 分钟工作坊

1. 10 分钟：规则讲解和 baseline 观察。
2. 15 分钟：最小生物 + 第一版策略。
3. 20 分钟：针对失败原因进行第二版策略。
4. 15 分钟：同学之间互相对战和观察。
5. 15 分钟：引入属性预算和策略取舍。
6. 15 分钟：写升级报告，说明下一轮想优化什么。

## 面向 Simulator 的迁移建议

如果 `Simulator` 项目也要支持学生用 vibe coding 创造对象，可以借鉴以下结构。

### 1. 定义一个窄 API

为学生对象定义一个小接口，例如：

```python
def decide(observation) -> action:
    ...
```

`observation` 应该稳定、有限、可解释；`action` 应该是意图，而不是直接改世界。

### 2. 保留统一规则裁判

仿真规则由引擎统一执行。学生可以请求移动、采集、建造、协作或繁殖，但是否成功由世界规则决定。

### 3. 提供 baseline 角色

准备 3-5 个行为不同、弱点明显的内置角色，作为学生比较和克制对象。

### 4. 加入运行时护栏

学生代码必须被安全调用。异常、非法返回值、超范围动作都应该降级为默认动作，并显示简洁诊断。

### 5. 把观察转成升级目标

界面或日志应该帮助回答：

- 我的对象为什么失败？
- 它做了哪些成功动作？
- 它和 baseline 的差距在哪里？
- 下一轮只改一个什么策略最值得？

### 6. 支持可重复实验

为了让讨论更严谨，后续应支持固定随机种子、多局平均、无渲染快速评估和训练历史输出。

## 可继续改进的方向

`swarm_arena` 后续可以继续增强这些能力，也可作为 `Simulator` 的设计目标参考：

1. 事件日志：记录吃食物、捕食、繁殖、死亡原因。
2. 单局报告：游戏结束后按物种输出表现摘要。
3. 学生升级卡片：把“现象、假设、改动、验证方式”结构化保存。
4. 对战配置：选择参赛物种、数量、地图大小和随机种子。
5. 快速评估：无界面运行多局，输出平均存活 tick、胜率和种群峰值。
6. 策略可视化：展示当前目标、逃跑触发、繁殖触发等调试信息。
7. LLM 辅助回顾：根据事件日志生成下一轮升级建议，但不直接替学生做完所有设计。

## 关键经验

这个项目最有价值的部分不是某个具体规则，而是它把学生创作压缩成一个清晰循环：

```text
提出生物设想 -> 编写 decide() -> 放入世界 -> 观察竞争结果 -> 讨论升级目标 -> 再迭代
```

对 `Simulator` 来说，最值得复用的是这个循环，以及支撑循环的工程边界：

- 简单入口。
- 有限感知。
- 意图动作。
- 统一裁判。
- 明确取舍。
- 可视反馈。
- 运行护栏。
- 可讨论的失败。

只要这些边界成立，学生就可以大胆使用 vibe coding，而老师仍能把课堂讨论拉回到模型、策略和证据上。
